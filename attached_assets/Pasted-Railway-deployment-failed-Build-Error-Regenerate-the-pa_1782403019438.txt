## Problem
The build fails at the npm ci step with ENOTFOUND package-firewall.replit.local. Railway's Railpack builder (Node.js provider) runs npm ci, which strictly uses the resolved URLs from package-lock.json. The lockfile contains Replit-internal URLs like http://package-firewall.replit.local/npm/tsx/-/tsx-4.22.4.tgz that are unreachable outside Replit. The failure stage is BUILD_IMAGE and the error is confirmed in build logs: "npm error network request to http://package-firewall.replit.local/npm/tsx/-/tsx-4.22.4.tgz failed, reason: getaddrinfo ENOTFOUND package-firewall.replit.local".

## Diagnosis
The package-lock.json was generated inside Replit's environment, which routes npm through an internal proxy (package-firewall.replit.local). This proxy hostname is baked into the resolved fields throughout the lockfile for many packages (confirmed by reading package-lock.json which shows hundreds of "resolved": "http://package-firewall.replit.local/npm/..." entries). While .npmrc sets registry=https://registry.npmjs.org/, npm ci ignores the registry config and uses the exact resolved URLs from the lockfile. Previous builds succeeded due to build cache; this build had a cache miss, triggering actual downloads that failed.

## Fix
1. **File:** package-lock.json (repo root)
2. **Current content:** Contains hundreds of "resolved": "http://package-firewall.replit.local/npm/..." entries throughout the file.
3. **Required change:** Regenerate the entire package-lock.json by running `npm install` in an environment outside of Replit (locally, in CI, or in any standard environment with internet access to npmjs.org). This will produce a lockfile with standard https://registry.npmjs.org/ URLs. Commit and push the updated package-lock.json.
4. **Why this fixes it:** With standard registry URLs, npm ci on Railway will be able to download all packages from the public npm registry without hitting Replit's internal proxy hostname.
